# HX-DataLock V1 Specification

HX-DataLock v1 is a Crypto Codec for converting Payload Bytes to and from portable encrypted Data Envelopes. It does not perform network I/O, database access, object storage access, cache management, synchronization, business object serialization, compression, or migration.

## Threat Model

Public Storage is hostile. Attackers may read, copy, delete, replay, or tamper with Data Envelopes and Public Key Documents.

The Write-only Sender receives only a Public Key Document. It can lock new Payload Bytes but cannot open existing Data Envelopes.

The User DataLock runs in a local user environment. It receives the full Keyring and the Master Password, unwraps the Read Key locally, and can open Data Envelopes. It may also lock new Payload Bytes using the Write Key already present in the full Keyring.

If a full Keyring leaks, security depends on the Master Password resisting an Offline Guessing Attack. Password attempt lockout is not a security control for v1 because leaked Keyrings can be copied and attacked offline.

## Algorithms

V1 uses:

- Write/Read key pair: X25519.
- Envelope key agreement: X25519 ephemeral sender key with recipient Write Key.
- Content key derivation: HKDF-SHA256, 32-byte output.
- Payload encryption: AES-256-GCM with 12-byte nonce and 16-byte tag.
- Read Key wrapping: scrypt-derived 32-byte wrapping key plus AES-256-GCM.

Default scrypt parameters:

- `N = 2^18`
- `r = 8`
- `p = 1`
- `keyLength = 32`

Implementations may allow lower scrypt parameters for tests, but generated Keyrings must record the actual parameters used.

## Master Password

The Master Password is a strict Unicode string. Before key derivation, SDKs must normalize it with Unicode NFC and then encode it as UTF-8.

SDKs must minimize Master Password residency:

- Do not store it in SDK object fields.
- Do not log it.
- Do not include it in thrown error messages.
- Do not write it into Keyrings, Public Key Documents, Data Envelopes, cache files, or environment output.
- Use it immediately for key derivation and then drop references.

Managed runtimes cannot guarantee complete memory zeroization. Memory clearing is best-effort only.

## Password Strength

SDKs expose a Password Strength Report before Keyring creation. Weak Master Passwords do not block creation in v1.

Report shape:

```ts
type PasswordStrengthReport = {
  level: "weak" | "fair" | "good" | "strong"
  allowed: true
  warnings: string[]
  suggestions: string[]
  estimatedEntropyBits?: number
}
```

Implementations should use mature password-estimation libraries where available. The exact estimator is not part of the Keyring or Data Envelope format, and exact scores do not need to match across languages. Shared tests should ensure obvious weak passwords are warned about and strong passphrases are not treated as weak.

## JSON Documents

V1 uses JSON for Keyrings, Public Key Documents, and Data Envelopes.

Writers should produce stable JSON:

- UTF-8 encoding.
- Two-space indentation.
- Documented field order.
- Trailing newline.
- Base64 for binary fields.

Readers must parse JSON by field and must not require field order to match writer output.

Creation Time fields are non-security metadata. They are useful for display and debugging, but they are not business timestamps, sync versions, or conflict-resolution fields.

## Keyring

A Keyring contains the Write Key and the encrypted Read Key. It is used only by User DataLock.

Required top-level fields:

```json
{
  "schema": "hxdl.keyring.v1",
  "createdAt": "2026-06-20T00:00:00.000Z",
  "publicWriteKey": {
    "alg": "X25519",
    "keyId": "x25519:<hash>",
    "spki": "<base64 DER SPKI>"
  },
  "encryptedReadKey": {
    "kdf": {
      "name": "scrypt",
      "salt": "<base64 32 bytes>",
      "N": 262144,
      "r": 8,
      "p": 1,
      "keyLength": 32
    },
    "aead": {
      "name": "AES-256-GCM",
      "nonce": "<base64 12 bytes>",
      "tag": "<base64 16 bytes>"
    },
    "ciphertext": "<base64 encrypted DER PKCS8 X25519 private key>"
  }
}
```

`publicWriteKey.keyId` is derived from the public Write Key. V1 uses:

```text
x25519:<first 22 chars of base64url(sha256(publicWriteKey.spki DER)) without padding>
```

The encrypted Read Key AAD must bind:

- Keyring schema.
- Write Key ID.
- Read Key wrapping algorithm identifiers.

Creation Time is not part of AAD.

## Public Key Document

A Public Key Document contains only the Write Key and non-secret metadata. It is the only supported input for Sender DataLock.

Required top-level fields:

```json
{
  "schema": "hxdl.publicKey.v1",
  "createdAt": "2026-06-20T00:00:00.000Z",
  "publicWriteKey": {
    "alg": "X25519",
    "keyId": "x25519:<hash>",
    "spki": "<base64 DER SPKI>"
  }
}
```

Write-only Senders must not receive a full Keyring, Master Password, encrypted Read Key, or any other Master Password-related material.

## Data Envelope

V1 supports Full Data Envelopes only. A Full Data Envelope encrypts one complete Payload Bytes value as a single authenticated payload.

Required top-level fields:

```json
{
  "schema": "hxdl.envelope.v1",
  "createdAt": "2026-06-20T00:00:00.000Z",
  "recipientKeyId": "x25519:<hash>",
  "alg": {
    "kem": "X25519",
    "kdf": "HKDF-SHA256",
    "aead": "AES-256-GCM"
  },
  "ephemeralPublicKey": "<base64 DER SPKI>",
  "hkdfSalt": "<base64 32 bytes>",
  "nonce": "<base64 12 bytes>",
  "tag": "<base64 16 bytes>",
  "ciphertext": "<base64 encrypted payload bytes>"
}
```

Envelope AAD must bind:

- Envelope schema.
- Recipient Key ID.
- Declared algorithm identifiers.

Creation Time is not part of AAD.

Data Envelopes target exactly one Recipient Key ID. Multi-recipient envelopes are not part of v1.

## Payload Bytes

The Crypto Codec accepts and returns bytes. Business formats such as JSON are encoded and decoded by the application before locking and after opening.

Text helpers must use strict UTF-8. Invalid UTF-8 must fail with `INVALID_UTF8`.

V1 does not compress Payload Bytes.

V1 does not pad Payload Bytes to hide Data Envelope size. Public Storage observers may learn approximate payload size, Creation Time, and Recipient Key ID. They do not learn Master Password length from v1 documents.

## File Limit

Local File Helpers are convenience APIs that read and write local files while producing or consuming Data Envelopes.

V1 supports single files up to 25 MB as Full Data Envelopes. Files larger than 25 MB must fail with `OVERSIZED_FILE`.

Chunked File Envelopes and stream APIs are deferred.

## SDK Surface

TypeScript and Python provide full SDKs:

- Keyring creation.
- Public Key Document export.
- Sender DataLock.
- User DataLock.
- Local File Helpers.
- CLI support.

Android Kotlin provides a modern Android client SDK with minimum target Android API 26+:

- User DataLock.
- `openBytes`.
- Local `lockBytes` using the Write Key from the full Keyring.

Kotlin does not provide Sender DataLock in v1.

### Sender DataLock

Sender DataLock accepts only a Public Key Document.

Required operations:

```ts
makeSenderDataLock(publicKeyDocument): SenderDataLock
sender.lockBytes(payloadBytes): DataEnvelope
sender.lockText(text): DataEnvelope
sender.lockFile(inputPath, outputPath): void
```

`lockText` encodes strict UTF-8. `lockFile` enforces the 25 MB limit.

Sender DataLock must not expose `openBytes`, `openText`, `openFile`, Read Key unwrapping, or Master Password input.

### User DataLock

User DataLock accepts a full Keyring and the Master Password.

Required operations:

```ts
makeUserDataLock(keyring, { masterPassword }): UserDataLock
user.openBytes(envelope): bytes
user.openText(envelope): string
user.openFile(inputPath, outputPath): void
user.lockBytes(payloadBytes): DataEnvelope
user.lockText(text): DataEnvelope
user.lockFile(inputPath, outputPath): void
user.close(): void
```

User DataLock caches the unwrapped Read Key for the object lifetime. It does not cache the Master Password. `close()` should drop references to cached key material, with managed-runtime memory clearing treated as best-effort.

## CLI

The CLI may read the Master Password from an explicit environment variable option. SDKs do not read environment variables themselves.

Interactive `init` must ask for the Master Password twice. Non-interactive environment-variable input skips confirmation.

Recommended CLI commands:

```text
hxdl init --keyring keyring.hxdl.json
hxdl export-public --keyring keyring.hxdl.json --out public-key.hxdl.json
hxdl lock --public-key public-key.hxdl.json --in plain.bin --out sealed.hxdl.json
hxdl open --keyring keyring.hxdl.json --in sealed.hxdl.json --out plain.bin
hxdl verify-keyring --keyring keyring.hxdl.json
hxdl verify-public --public-key public-key.hxdl.json
hxdl bench
```

Write-only Sender environments should not define Master Password environment variables.

## Error Codes

V1 exposes stable DataLock Error Codes:

- `INVALID_KEYRING`
- `INVALID_PUBLIC_KEY_DOCUMENT`
- `WRONG_MASTER_PASSWORD_OR_TAMPERED_KEYRING`
- `ENVELOPE_RECIPIENT_MISMATCH`
- `TAMPERED_ENVELOPE`
- `UNSUPPORTED_SCHEMA`
- `OVERSIZED_FILE`
- `INVALID_UTF8`
- `UNSUPPORTED_ALGORITHM`

Weak Master Passwords are reported through Password Strength Reports, not error codes.

## Performance

V1 performance documentation covers Full Data Envelopes up to 25 MB.

Benchmarks should measure:

- Keyring unlock time.
- `lockBytes` and `openBytes` for 1 MB, 10 MB, and 25 MB.
- `lockFile` and `openFile` for 1 MB, 10 MB, and 25 MB.

V1 does not publish performance claims for 100 MB, 1 GB, or 10 GB payloads because those require the deferred Chunked File Envelope design.

## Cross-language Compatibility

Required compatibility tests:

- TypeScript creates Keyring, Python verifies and opens.
- Python creates Keyring, TypeScript verifies and opens.
- TypeScript locks, Python opens.
- Python locks, TypeScript opens.
- Android Kotlin opens TypeScript and Python Data Envelopes.
- Android Kotlin locks locally, TypeScript and Python open.
- Public Key Document exported by one full SDK is accepted by the other full SDK's Sender DataLock.
- Wrong Master Password fails with `WRONG_MASTER_PASSWORD_OR_TAMPERED_KEYRING`.
- Tampered envelope fails with `TAMPERED_ENVELOPE`.
- Recipient mismatch fails with `ENVELOPE_RECIPIENT_MISMATCH`.

## Deferred

V1 deliberately does not support:

- Chunked File Envelopes.
- Streaming APIs.
- Files larger than 25 MB.
- Key rotation.
- Master Password change.
- Multi-recipient envelopes.
- Platform Keychain or Keystore integration.
- Network, database, cloud drive, local cache, or sync adapters.
- Built-in compression.
- Payload length padding.
- Password attempt lockout.
