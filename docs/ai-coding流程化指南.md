# AI 编程流程指南

本文件介绍项目中可用的 AI 工程技能及其推荐使用流程。

## 技能分类

### 使用 Issue Tracker 配置的技能(`docs/agents/issue-tracker.md`)

| 技能 | 作用 | 调用方式 |
|---|---|---|
| **`qa`** | 交互式 QA 会话。用户描述 bug, 助手探索代码库获取领域上下文, 然后将问题发布到 `.scratch/<feature>/` 下。Issue 用领域语言编写, 不引用具体文件/行号, 保证重构后仍然可读。 | 用户说 "QA session" 或 "报个 bug" 等 |
| **`review`** | 双轴代码审查。从固定点(commit/branch)到 HEAD 的 diff, 同时跑两个子代理: 一个检查编码标准合规性, 一个检查规格实现完整性。Issue tracker 配置用于查找 commit message 中引用的原始 issue。 | `/review` 或 "review since X" |

### 使用 Triage 标签配置的技能(`docs/agents/triage-labels.md`)

| 技能 | 作用 | 调用方式 |
|---|---|---|
| **`review`** | (同上)审查发现问题后, 可按 triage 标签(待评估/等待补充/可供智能处理/需要人工/不予处理)分类提交 issue。 | `/review` 或 "review since X" |

### 使用 Domain Docs 配置的技能(`docs/agents/domain.md`)

| 技能 | 作用 | 调用方式 |
|---|---|---|
| **`diagnosing-bugs`** | 结构化排障六步法: ①建立反馈循环 → ②复现+最小化 → ③假设 → ④打点 → ⑤修复+回归测试 → ⑥清理+复盘。探索代码库时读取 `CONTEXT.md` 理解模块关系, 检查相关 ADR。核心原则: 没有红色反馈循环, 不进假设阶段。 | 用户说 "diagnose" / "debug this" 或描述 bug |
| **`tdd`** | 红-绿-重构循环。探索代码库时读取 `CONTEXT.md` 确保测试命名和接口词汇与领域语言一致, 并尊重相关 ADR。强调垂直切片(一次一个测试→实现), 反对先批量写所有测试再写代码的水平切片。 | 用户说 "red-green-refactor" 或 "写测试" |
| **`domain-modeling`** | 主动构建和打磨领域模型。用于定义术语表(写入 `CONTEXT.md`)、记录架构决策(写入 `docs/adr/`)。仅在三个条件同时满足时才建议创建 ADR: 难以逆转、缺少上下文时令人费解、经过真实权衡。 | 用户说 "定义术语" 或讨论领域概念时 |

### 不依赖配置文件、独立运行的技能

| 技能 | 作用 | 调用方式 |
|---|---|---|
| **`codebase-design`** | 为设计深层模块提供共享词汇(模块、接口、深度、缝合点、适配器、杠杆、局部性)。核心理念: 小接口 + 大实现 = 深层模块; 避免大接口 + 薄实现的浅层模块。 | 讨论模块设计/接口设计时 |
| **`grilling`** | 对计划或设计进行审问式访谈。逐一提问每个设计分支, 逐个解决决策间的依赖关系。每个问题附带推荐答案, 一次只问一个, 等用户反馈后再继续。 | 用户说 "grill this plan" 或需要审问方案时 |

## 推荐工作流程

### 场景 1: 收到一个 bug

```
用户描述 bug
       ↓
1. diagnosing-bugs  →  六步排障: 先建反馈循环, 再假设验证
       ↓
   bug 修复完成
       ↓
2. review           →  双轴审查: 标准合规 + 规格完整
       ↓
   如需讨论架构改进
       ↓
3. codebase-design  →  讨论深层模块设计
```

### 场景 2: 开发新功能

```
需求讨论
   ↓
1. grilling         →  审问式访谈, 澄清所有细节
   ↓
2. domain-modeling  →  定义术语, 必要时写 ADR
   ↓
3. tdd              →  红-绿-重构, 垂直切片推进
   ↓
4. review           →  双轴审查
```

### 场景 3: 架构重构/改进

```
1. codebase-design  →  用深层模块词汇分析现状
   ↓
2. domain-modeling  →  梳理术语, 记录关键架构决策
   ↓
3. tdd              →  测试驱动重构
   ↓
4. review           →  双轴审查
```

### 场景 4: QA 测试回合

```
用户测试、发现多个问题
       ↓
1. qa               →  逐个收集问题, 发布到 issue tracker
       ↓
   后续可对每个 issue
       ↓
2. diagnosing-bugs  →  逐个排查修复
```

## 各技能详细说明

### diagnosing-bugs — 结构化排障

**六阶段流程**(不可跳步):

1. **建立反馈循环** — 最重要的阶段。找一个能复现 bug 的命令(测试、curl、CLI 等), 必须能红(失败)/绿(通过)。没有红循环不进下一阶段。
2. **复现 + 最小化** — 确保是用户描述的同一个 bug, 然后裁剪到最小复现场景。
3. **假设** — 生成 3-5 个可证伪的假设, 格式: "如果 X 是原因, 那么改 Y 会让 bug 消失/变严重"。
4. **打点** — 用 debugger 或带标签的日志(`[DEBUG-xxx]`)逐一验证假设。一次只改一个变量。
5. **修复 + 回归测试** — 在有正确缝合点时先写回归测试再造。修复完重新跑原始复现命令。
6. **清理 + 复盘** — 清理所有 debug 代码, 总结正确假设。如有架构级的改进建议, 移交 `codebase-design`。

### tdd — 测试驱动开发

**核心原则**: 测试通过公共接口验证行为, 不绑定实现细节。好的测试重构不会破坏。

**禁止水平切片**: 不要先写所有测试再写所有代码。要垂直切片: 一个测试 → 一个实现 → 重复。

```
错误(水平切片):
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确(垂直切片):
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  ...
```

### codebase-design — 深层模块设计

**核心词汇**:

| 术语 | 定义 |
|---|---|
| **模块** | 任何有接口和实现的东西(函数、类、包) |
| **接口** | 调用者需要知道的全部信息(类型、不变量、错误模式、性能特征) |
| **实现** | 模块内部的代码体 |
| **深度** | 接口小但行为多 = 深层; 接口复杂但实现薄 = 浅层 |
| **缝合点** | 接口存在的位置, 可以不编辑该位置就能改行为 |
| **适配器** | 在缝合点满足接口的具体实现 |
| **杠杆** | 调用者从深度获得的收益: 一份实现服务 N 个调用点 |
| **局部性** | 维护者从深度获得的收益: 修改、bug、知识集中一处 |

**设计检查项**:
- 删除测试: 删掉这个模块, 复杂性会消失(透传)还是在 N 处重现(有价值)?
- 接口即测试面: 调用者和测试穿过同一个缝合点
- 一个适配器是假设的缝合点, 两个适配器才是真实的

### review — 双轴审查

两轴独立并行:
- **标准轴**: 代码是否符合编码规范? 引用规范文件+具体规则。
- **规格轴**: 代码是否忠实实现了 issue/PRD 的要求? 引用规格原文。

两轴分开报告, 不合并排序——防止一轴掩盖另一轴。

### domain-modeling — 领域建模

**何时创建 CONTEXT.md**: 第一个领域术语被确定时。

**何时创建 ADR**: 同时满足以下三点:
1. 难以逆转
2. 缺少上下文时令人费解
3. 经过真实权衡(有真正的替代方案)

**CONTEXT.md 规则**: 纯术语表, 不含实现细节、不做事先约定、不做笔记。

### qa — QA 会话

对每个用户报告的问题:
1. 倾听 + 最多 2-3 个澄清问题
2. 后台探索代码库(学习领域语言, 不找修复方案)
3. 判断是单 issue 还是需拆分为多个
4. 发布 issue(使用领域语言, 不引用文件路径/行号)

### grilling — 审问式访谈

对计划逐层深入提问, 每次只问一个问题。每个问题提供推荐答案。遇到代码库能回答的问题时, 直接探索代码库替代提问。

## Triage 标签说明

使用中文标签, 五个状态:

| 中文标签 | 含义 |
|---|---|
| `待评估` | 维护者需要评估此 issue |
| `等待补充` | 等待报告人提供更多信息 |
| `可供智能处理` | 规格完备, 可供 AI 代理处理 |
| `需要人工` | 需要人工实现 |
| `不予处理` | 不予处理 |
